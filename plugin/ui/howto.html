How to Use Claude-Mem

Persistent memory that makes Claude smarter across sessions. Just talk naturally - Claude remembers your projects, decisions, and context.

0 MCP Quick Setup

Get your agent connected to claude-mem in 3 steps. The MCP server lets any Claude instance search and store memories.

1 Verify Worker is Running 

# Check the worker is responding
curl http://localhost:37777/api/readiness

# Expected: {"ready":true}

2 Add MCP Server to Claude Settings

Edit ~/.claude/settings.json and add the claude-mem MCP server: 

{
  "mcpServers": {
    "claude-mem": {
      "type": "stdio",
      "command": "node",
      "args": ["/path/to/plugin/scripts/mcp-server.cjs"],
      "env": {
        "CLAUDE_MEM_WORKER_URL": "http://localhost:37777"
      }
    }
  }
}

Tip: Replace /path/to/plugin with your actual plugin path (e.g., ~/.claude/plugins/marketplaces/thedotmack/claude-mem)

3 Restart Claude Code

Restart Claude Code to load the MCP server. You should see claude-mem in your available tools.

Prompt to Give Your Agent

Copy this prompt and give it to your Claude agent to connect to the memory system: 

You have access to claude-mem, a persistent memory system.

Available MCP tools:
- search: Search past observations (query, limit, project, type, dateStart, dateEnd)
- timeline: Get context around a specific observation (anchor or query)
- get_observations: Fetch full details for specific IDs

3-layer workflow for efficient memory access:
1. search(query) ‚Üí Get index with IDs (~50-100 tokens/result)
2. timeline(anchor=ID) ‚Üí Get context around interesting results
3. get_observations([IDs]) ‚Üí Fetch full details ONLY for filtered IDs

Examples:
- "What bugs have we fixed?" ‚Üí search for type:bugfix
- "What did we work on yesterday?" ‚Üí search with dateStart
- "Tell me about authentication changes" ‚Üí search query="authentication"

Always use search first to find relevant IDs, then fetch details only for what you need.

Verify It's Working

Ask your agent: "Search my memory for recent work"
If configured correctly, it will use the MCP tools to search your observations.

1 How It Works 

Your Session
    |
    |--> Tool Usage (Read, Write, Bash, etc.)
    |         |
    |         v
    |    Hooks capture observations
    |         |
    |         v
    |    Worker compresses & stores
    |         |
    |         v
    +--> SQLite + Chroma (semantic search)

Next Session
    |
    |--> Context auto-injected from last 10 sessions
    +--> MCP tools available for deeper searches

Automatic: Every tool call is captured, compressed into an "observation", and stored. Next session, relevant context is injected automatically.

4 Privacy Control

Wrap content in <private> tags to exclude from storage: 

<private>
API_KEY=sk-proj-abc123
Database host: internal-db.company.com
</private>

Help me debug this connection error

Claude sees it during the session, but it won't be stored in memory.

6 What Gets Stored

Captured

  • - Files read / written
  • - Tool inputs / outputs
  • - Decisions made
  • - Bug fixes & features
  • - Key learnings

NOT Captured

  • - Your full prompts (only summaries)
  • - Claude's full responses
  • - Anything in <private> tags

7 Multi-Agent Setup

Multiple Claude instances can share memories with fine-grained visibility controls. Perfect for teams or running multiple agents on the same project.

1 Register an Agent 

curl -X POST http://localhost:37777/api/agents/register \
  -H "Content-Type: application/json" \
  -d '{"id": "agent1@laptop", "department": "backend"}'

# Response includes your API key (save it!):
# {"agent": {...}, "apiKey": "cm_abc123..."}

2 Verify the Agent 

curl -X POST http://localhost:37777/api/agents/verify \
  -H "Content-Type: application/json" \
  -d '{"id": "agent1@laptop", "apiKey": "cm_abc123..."}'

3 Use in Requests 

curl http://localhost:37777/api/search?q=authentication \
  -H "Authorization: Bearer cm_abc123..."

Visibility Levels

private Only you can see
department Same department sees
project All agents see (default)
public Visible globally

Same Agent, Multiple Projects

Projects are identified by git remote URL. The same agent can work across different repos: 

# Each project is automatically identified:
github.com/you/project-a  ‚Üê Agent works here
github.com/you/project-b  ‚Üê Same agent, different memories
github.com/team/shared    ‚Üê Team project, shared memories

Key Management

  • ‚Ä¢ Keys expire after 90 days by default
  • ‚Ä¢ 5 failed attempts triggers a 5-minute lockout
  • ‚Ä¢ Use /api/agents/rotate-key to get a new key
  • ‚Ä¢ Use /api/agents/me to check key status
